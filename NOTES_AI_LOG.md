# AI build log — where AI helped, where the human overrode

Running notes kept during the build, per the case's README requirement. Newest at bottom.

## Environment
- **AI:** Set up Python 3.12 via `uv` on a machine with no Python installed; pinned deps in `requirements.txt` (numpy/pandas/scikit-learn/streamlit/plotly/matplotlib). Deliverable stays plain-pip compatible.

## Judgment calls made by AI (flagged for human review)
1. **Scale-model economics.** The spec pins the dataset at ~270K orders (≈50 orders/store/day) so it runs on a laptop, but real Solstice stores at $140M/60 stores do ~10–15× that volume. Decision: treat the simulation as a *scale model* — all labor **percentages and ratios** are computed natively inside the sim (they transfer across scale), and dollar impacts are reported both at sim scale and applied to the stated $140M fleet P&L with an explicit, labeled scale factor. No hidden inflation.
2. **Timestamps.** Toast returns `openedDate` in UTC (`+0000`) and you localize via the restaurant's timezone. The generator emits true UTC assuming all stores at a fixed UTC-6 offset (DST ignored, documented); the adapter converts back using the same constant and buckets dayparts on *local* time. `businessDate` (Toast's own operating-date concept) carries the local date — which is exactly why Toast has that field.
3. **Square/slow-store overlap.** The 15 Square stores and the 15 slow-tier stores are assigned independently (random overlap ≈ 3–4 stores). Making them the same set would inject a confound ("your labor problem is just a POS problem") that a skeptical reader would rightly attack.
4. **Shift shapes.** Sim-scale stores run few labor-hours/day, so generated time entries are 1–3h blocks placed inside daypart windows (part-time-block pattern). The adapter allocates entry time to dayparts by timestamp overlap, so it would handle arbitrary multi-daypart shifts from the real API unchanged.

## Strategy review after Checkpoint 1 (human-driven, AI-researched)
5. **Human asked the right product question:** "does Toast already do people-to-shift scheduling?" AI research confirmed yes — Scheduling powered by Sling (free Lite tier; Pro adds trailing-history sales projections and accepts imported projections); Square has Shifts. Consequence: we do NOT build assignment UI; we produce feasible staffing *targets* that feed the scheduler managers already use. Repositioned the artifact accordingly.
6. **Human added labor feasibility constraints** the AI's first design lacked: minimum 3h shifts, ≥20h/week per employee with ~35h preferred (turnover mitigation), no split shifts. These became the constraint layer in labor.py — and the roster-resizing framing (fewer-but-fuller schedules via attrition) came out of that discussion.

## Forecast calibration (Checkpoint 2) — two honest iterations, logged
7. **First run failed the gate honestly:** RF beat naive by only +3.4%, and the DOW-aware baseline was *worse* than naive. Diagnosis: the synthetic world's systematic signal (DOW ±10%, rare weather effects) was too small relative to Poisson noise at ~50 orders/day — every forecaster degenerated to a trailing average, and there was nothing for ML to add. This is the generator-miscalibration failure mode BUILD_PLAN §3.4 warns about.
8. **Fix was calibration toward realism, not toward the model:** weekend swings widened to realistic cafe amplitudes (Sat 1.5× cafes, 0.7× drive-thrus — commuter vs brunch), graded rain suppression up to 30%, graded temperature response, brunch-shifted weekend dayparts. RF hyperparams smoothed (min_samples_leaf 5→15 — small leaves were fitting Poisson noise). Added one standard feature (DOW index = trailing same-DOW / trailing overall mean). The model never sees generator internals — only calendar, weather, store attributes, and its own trailing history.
9. **Final ladder (held-out days 71–90):** naive 30.2% wMAPE → DOW-aware 26.9% → RF 24.4%. RF ≈ +19–20% vs naive (varies ±1pt across regenerations — we report the honest number, not the best draw), +9.4% vs the DOW-aware spreadsheet. We report the *ladder*, not just the headline: a smarter spreadsheet gets you half the win; ML earns the rest via weather/trend/payday and cross-store pooling. Weighted MAPE (sum|err|/sum actual) chosen because plain MAPE explodes on near-zero evening dayparts.

## Labor layer (Checkpoint 3) — where AI proposed and human constraints reshaped it
10. **Human overrode the AI's unconstrained plan twice.** (a) The human required real employment constraints — 3h minimum shifts, ≥20h/week, ~35h preferred, no split shifts — which the AI's first design lacked; encoding them cut the "opportunity" from a too-clean 96% of the theoretical gap to an honest 62% ($3.42M of $5.5M). (b) The AI's first output recommended one store down to 24.5% labor — below anything Solstice has demonstrated; the fix was a 27% plan floor (1pt below target = edge of demonstrated operations).
11. **AI caught its own noise problem:** day-level "do not cut" flags were Poisson-noise-driven and scattered across the wrong tier. Split into (a) day-level guardrail that blocks cuts, (b) window-average *systematic* understaffing flag for the narrative. Result: all 15 slow stores flag understaffed at morning peak — the case's core insight, recovered from data the model never saw labeled.
12. **Service standard is calibrated, not assumed:** 3.95 labor-minutes per prep-minute, derived from flagship stores (the tier already at ~target) on the training window. The debrief answer to "where did your staffing standard come from?" is: "from your own best stores."

## App + chatbot (Checkpoint 4) — human product judgment, AI implementation
13. **Human set the product frame three times:** (a) organize the app by *audience* — store manager first, leadership second — not by analytics topic; (b) the chatbot serves the STORE MANAGER (their staffing questions, their overrides), not a generic Q&A; (c) API keys are backend-only ("a store manager should never see an API key") — the AI's original sidebar key input was removed on human instruction, replaced by gitignored secrets/env with graceful degradation.
14. **AI kept the LLM out of the numbers:** the "why" rationale a manager sees is counterfactual re-prediction (arithmetic), and the chatbot is grounded in the computed plan with explicit rules: never invent numbers, never cut flagged dayparts, route disagreement to reason-coded overrides. Live-tested: grounding, what-if arithmetic, and the override routing all behaved as designed.
15. **AI caught two of its own display bugs during browser verification:** sim-scale numbers leaking into fleet-scale UI (rationale + chat context), and Streamlit rendering dollar amounts as LaTeX. Both fixed and re-verified live.
16. **Human caught a unit-clarity trap the AI missed:** the staffing table showed decimal "staff" counts (6.4 people?) — correct as *average concurrent headcount*, but misreadable as fractional scheduling. Fix: columns relabeled "avg on floor", the Change column now speaks only in whole 3-hour shifts, and the caption states the two-currency rule — the model plans in hours, acts in whole shifts. If a table confuses the person who built the tool, it will confuse a store manager.
